# LifePlan Navigator インシデント対応訓練計画書

**文書番号**: LPN-IRP-004
**バージョン**: 1.0
**作成日**: 2025-12-11
**承認者**: CSIRT Team Leader
**分類**: 社外秘

---

## 1. 概要

### 1.1 目的
インシデント対応能力の維持・向上を目的とした体系的な訓練プログラムを実施し、実際のインシデント発生時に迅速かつ適切に対応できる体制を確立する。

### 1.2 訓練の種類

| 種類 | 概要 | 頻度 | 所要時間 |
|------|------|------|----------|
| **テーブルトップ演習** | 座学形式でのシナリオ討議 | 四半期毎 | 2-3時間 |
| **実機演習** | 実環境またはテスト環境での対応訓練 | 半期毎 | 4-8時間 |
| **フルスケール演習** | 全社参加の総合訓練 | 年1回 | 1-2日 |
| **個別スキル訓練** | 特定スキルの習得訓練 | 随時 | 1-2時間 |

### 1.3 参加対象者

| 層 | 対象者 | 参加必須訓練 |
|----|--------|-------------|
| **経営層** | CEO, CISO, CLO, CFO, CMO | テーブルトップ、フルスケール |
| **統括層** | CSIRT Team Leader, Auditor, CTI Analyst | 全訓練 |
| **実行層** | SOC Analyst, CSIRT Engineer, Network Engineer, App Engineer | 全訓練 |

## 2. テーブルトップ演習（TTX）

### 2.1 演習概要

**形式**: 会議室での座学形式、シナリオベースの討議
**進行**: ファシリテーター（CSIRT Team Leader or 外部講師）
**記録**: 議事録・改善点の記録

### 2.2 年間スケジュール

| 四半期 | シナリオ | 主要参加者 | 重点ポイント |
|--------|----------|----------|-------------|
| Q1 | データ漏洩 | 全員 | 法的報告義務、顧客対応 |
| Q2 | ランサムウェア | 技術チーム中心 | 封じ込め、復旧判断 |
| Q3 | DDoS攻撃 | 技術チーム中心 | 緩和策、外部連携 |
| Q4 | 内部不正 | 経営層+HR | 調査手法、法的対応 |

### 2.3 TTX実施手順

#### 2.3.1 事前準備（演習2週間前）

```
□ シナリオ作成
  - 想定状況の設定
  - タイムラインの作成
  - インジェクト（追加情報）の準備

□ 参加者調整
  - 日程調整
  - 役割分担の決定
  - 事前資料の配布（必要に応じて）

□ 会場・機材準備
  - 会議室予約
  - ホワイトボード/モニター
  - 記録担当の手配
```

#### 2.3.2 演習当日の流れ

```
【導入】(15分)
├── 演習目的の説明
├── ルール説明（正解はない、自由に発言）
└── 役割確認

【シナリオ提示・討議】(90-120分)
├── Phase 1: 検知・初動 (30分)
│   └── 「このアラートを受けて、まず何をしますか？」
├── Phase 2: 封じ込め・対応 (30分)
│   └── 「影響が拡大しています。どう対処しますか？」
├── Phase 3: 復旧・報告 (30分)
│   └── 「復旧判断の基準は？誰に何を報告しますか？」
└── インジェクト対応 (各フェーズ随時)
    └── 「メディアから問い合わせが入りました...」

【振り返り】(30分)
├── 良かった点の共有
├── 改善が必要な点の討議
└── アクションアイテムの決定
```

### 2.4 TTXシナリオ例

#### シナリオ例1: 大規模データ漏洩

```
【背景】
本日15:30、セキュリティ監視チームがDBサーバーへの大量アクセスを検知。
調査の結果、過去2週間にわたり、約50万件の顧客情報（氏名、メール、
金融資産情報）が外部に送信された可能性。

【インジェクト1】(30分後)
攻撃者と思われるアカウントから「データを公開されたくなければ...」
というメールが届く。

【インジェクト2】(60分後)
セキュリティブログに「LifePlan Navigator」のデータと称する
リストが掲載される。

【インジェクト3】(90分後)
記者から「情報漏洩について確認したい」と電話。

【討議ポイント】
- 初動で何を優先するか？
- 身代金要求にどう対応するか？
- 個人情報保護委員会への報告タイミングは？
- 顧客への通知方法は？
- メディア対応の方針は？
```

#### シナリオ例2: ランサムウェア感染

```
【背景】
月曜日朝9:00、出社した従業員から「PCが起動しない」「ファイルが
開けない」との報告が相次ぐ。IT部門が確認したところ、
ランサムウェアによる暗号化を確認。

【インジェクト1】(20分後)
感染端末が30台に拡大。業務システムのファイルサーバーも暗号化。

【インジェクト2】(45分後)
直近のバックアップが3日前であることが判明。

【インジェクト3】(70分後)
身代金要求額は500万円。支払い期限は48時間。

【討議ポイント】
- 感染拡大防止のための優先アクションは？
- バックアップからの復旧か、身代金支払いか？
- 業務継続の代替手段は？
- 法執行機関への連絡は？
```

## 3. 実機演習

### 3.1 演習概要

**形式**: テスト環境または本番環境（制御下）での実践的訓練
**特徴**: 実際のツール・システムを使用した対応訓練

### 3.2 演習環境

| 環境 | 用途 | 特徴 |
|------|------|------|
| **サンドボックス環境** | マルウェア解析訓練 | 本番から完全隔離 |
| **ステージング環境** | 復旧・封じ込め訓練 | 本番相当の構成 |
| **本番環境（制御下）** | フルスケール演習 | 事前通知の上で実施 |

### 3.3 年間スケジュール

| 時期 | 演習名 | 対象 | 内容 |
|------|--------|------|------|
| 上期 | **フォレンジック演習** | CSIRT Eng, SOC | メモリ・ディスク解析 |
| 下期 | **復旧演習** | 全技術チーム | バックアップからの復旧 |

### 3.4 実機演習プログラム

#### 3.4.1 フォレンジック演習

**目的**: 証拠収集・分析スキルの向上

```
【事前準備】
□ 演習用マルウェア検体の準備（無害化済み）
□ 感染環境のスナップショット作成
□ フォレンジックツールの準備

【演習内容】(4時間)
1. メモリイメージ取得 (30分)
   - Volatilityを使用したメモリダンプ
   - プロセス・ネットワーク接続の確認

2. ディスクイメージ取得 (30分)
   - ddによるイメージ取得
   - ハッシュ値の記録（Chain of Custody）

3. タイムライン分析 (60分)
   - ファイルシステムタイムライン作成
   - 不審なファイル・イベントの特定

4. マルウェア解析（静的）(60分)
   - ファイルハッシュ確認
   - 文字列抽出・分析

5. レポート作成 (60分)
   - 調査結果の文書化
   - IOCの抽出

【評価ポイント】
- 証拠の完全性維持
- 分析の正確性
- 文書化の品質
```

#### 3.4.2 復旧演習

**目的**: バックアップからの復旧手順の確認・習熟

```
【事前準備】
□ ステージング環境の準備
□ 各種バックアップの確認
□ 復旧手順書の最新化

【演習内容】(8時間)
1. 状況設定・チームブリーフィング (30分)
   - ランサムウェア被害を想定
   - 復旧対象システムの確認

2. バックアップ検証 (60分)
   - バックアップの整合性確認
   - 感染していないことの確認

3. 復旧優先順位の決定 (30分)
   - BCP/BCMに基づく優先順位付け

4. 復旧作業 (4時間)
   - データベース復旧
   - アプリケーションサーバー復旧
   - ネットワーク設定復旧

5. 動作確認 (60分)
   - 機能テスト
   - データ整合性確認

6. 振り返り (60分)
   - 問題点の洗い出し
   - 手順書の改善

【評価ポイント】
- RTO（目標復旧時間）の達成
- RPO（目標復旧時点）の達成
- 手順書の有効性
```

## 4. フルスケール演習

### 4.1 演習概要

**形式**: 全社参加の大規模総合訓練
**特徴**: 実際に近い条件下での組織全体の対応訓練
**頻度**: 年1回

### 4.2 演習計画

#### 4.2.1 演習シナリオ例

```
【演習名】Operation Phoenix 2025

【想定状況】
標的型攻撃により、段階的にシステムが侵害される複合シナリオ

【タイムライン】
Day 1 - 09:00  : フィッシングメールによる初期侵入
Day 1 - 12:00  : 内部偵察・横展開の検知
Day 1 - 15:00  : 特権アカウント侵害
Day 1 - 18:00  : データ窃取の痕跡発見
Day 2 - 09:00  : ランサムウェア展開
Day 2 - 12:00  : 身代金要求・メディア報道
Day 2 - 15:00  : 復旧開始
Day 2 - 18:00  : 演習終了・ホットウォッシュ
```

#### 4.2.2 参加者・役割

| チーム | メンバー | 役割 |
|--------|----------|------|
| **Red Team** | 外部ベンダー | 攻撃シミュレーション |
| **Blue Team** | 全技術チーム | 検知・対応 |
| **White Team** | 演習事務局 | 進行管理・評価 |
| **経営チーム** | 経営層 | 意思決定・広報対応 |
| **Observer** | 監査・外部評価者 | 客観的評価 |

#### 4.2.3 実施手順

```
【準備フェーズ】(演習3ヶ月前〜)
Week 1-4:  演習計画策定、シナリオ作成
Week 5-8:  環境準備、参加者調整
Week 9-12: リハーサル、最終調整

【実施フェーズ】(演習当日)
Day 1: 攻撃シミュレーション・検知・初動対応
Day 2: 封じ込め・復旧・経営対応

【事後フェーズ】(演習後2週間以内)
Week 1: ホットウォッシュ（直後振り返り）
Week 2: 詳細評価、改善レポート作成
```

### 4.3 評価基準

| 評価項目 | 評価基準 | 配点 |
|----------|----------|------|
| **検知能力** | 検知までの時間、検知率 | 20点 |

| **意思決定** | 判断の適切性、迅速性 | 15点 |
| **復旧** | 復旧時間、完全性 | 15点 |

## 付録

### 付録A: TTXファシリテーションガイド

```
【進行のポイント】
1. 正解はないことを強調
2. 全員が発言できる雰囲気作り
3. 役職に関係なく意見を求める
4. 議論が停滞したらインジェクト投入
5. 批判ではなく改善に焦点

【よくある質問への対応】
Q: 「そんな状況はありえない」
A: 「想定外の事態にどう対応するかを考えましょう」

Q: 「手順書にないのでわからない」
A: 「手順書がない状況でどう判断しますか？」
```
