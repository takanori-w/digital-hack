# LifePlan Navigator インシデント対応手順書

**文書番号**: LPN-IRP-001
**バージョン**: 1.0
**作成日**: 2025-12-11
**承認者**: CSIRT Team Leader
**分類**: 機密

---

## 1. 概要

### 1.1 目的
本手順書は、LifePlan Navigatorにおけるセキュリティインシデント発生時の対応プロセスを定義し、迅速かつ適切な対応を実現することを目的とする。

### 1.2 適用範囲
- LifePlan Navigatorシステム全体
- 関連するインフラストラクチャ
- ユーザーデータ（個人情報・金融情報を含む）

### 1.3 準拠フレームワーク
- **SANS PICERL**: Preparation, Identification, Containment, Eradication, Recovery, Lessons Learned
- **NIST SP 800-61**: Computer Security Incident Handling Guide
- **個人情報保護法**: 漏えい等報告義務

## 2. インシデント分類

### 2.1 重大度レベル

| レベル | 名称 | 定義 | 対応時間 |
|--------|------|------|----------|
| **P0** | Critical | 個人情報大量漏洩、サービス全停止、ランサムウェア感染 | **即時対応** |
| **P1** | High | 限定的データ漏洩、主要機能停止、標的型攻撃検知 | **4時間以内** |
| **P2** | Medium | 不正アクセス試行、軽微な脆弱性悪用、パフォーマンス低下 | **24時間以内** |
| **P3** | Low | ポートスキャン、スパム、ポリシー違反 | **1週間以内** |

### 2.2 インシデントカテゴリ

| カテゴリ | 説明 | 主な例 |
|----------|------|--------|
| **DATA** | データ関連 | 情報漏洩、データ改ざん、不正アクセス |
| **AVAIL** | 可用性関連 | DDoS、システム障害、サービス停止 |
| **MALWARE** | マルウェア関連 | ランサムウェア、トロイの木馬、バックドア |
| **INSIDER** | 内部不正 | 従業員による不正、権限悪用 |
| **VULN** | 脆弱性悪用 | SQLi、XSS、認証バイパス |

## 4. SANS PIECRLプロセス

### 4.1 Phase 1: Preparation（準備）

#### 4.1.1 事前準備項目

- [ ] インシデント対応チームの編成・訓練
- [ ] 連絡体制の整備（緊急連絡網）
- [ ] 対応ツールの準備
  - フォレンジックツール（Volatility, Autopsy）
  - ログ分析ツール（Splunk, ELK Stack）
  - ネットワーク分析ツール（Wireshark, tcpdump）
- [ ] バックアップ・復旧手順の確認
- [ ] 外部連絡先リストの整備
  - 警察（サイバー犯罪対策課）
  - 個人情報保護委員会
  - セキュリティベンダー
  - 法律事務所

#### 4.1.2 必要資材

| 区分 | 項目 | 保管場所 |
|------|------|----------|
| ハードウェア | フォレンジック用PC | セキュリティルーム |
| ハードウェア | 書き込み防止装置 | セキュリティルーム |
| ハードウェア | 予備ストレージ | データセンター |
| ソフトウェア | イメージ取得ツール | フォレンジックPC |
| 文書 | 対応手順書（本書） | 社内Wiki + 印刷版 |
| 文書 | 連絡先リスト | 社内Wiki + 印刷版 |

### 4.2 Phase 2: Identification（識別）

#### 4.2.1 検知ソース

| ソース | 監視項目 | 担当 |
|--------|----------|------|
| SIEM | 異常ログパターン、相関分析 | SOC Analyst |
| IDS/IPS | 不正通信、攻撃シグネチャ | Network Engineer |
| WAF | Webアプリケーション攻撃 | SOC Analyst |
| EDR | エンドポイント脅威 | CSIRT Engineer |
| ユーザー報告 | 不審な挙動、フィッシング | サポートデスク |

#### 4.2.2 初動確認項目

```
□ アラートの確認と検証
  - 誤検知の可能性確認
  - 影響範囲の初期評価

□ インシデント判定
  - カテゴリ分類
  - 重大度レベル判定

□ 初期情報収集
  - 発生日時
  - 影響を受けたシステム/データ
  - 攻撃元情報（IP、手法）

□ エスカレーション判断
  - 該当するエスカレーションパスの選択
```

#### 4.2.3 インシデント判定フロー

```
[アラート検知]
      │
      ↓
[誤検知確認] ──Yes──→ [クローズ・記録]
      │
     No
      ↓
[影響範囲確認]
      │
      ├── 個人情報漏洩の可能性 ──→ P0/P1
      ├── サービス停止 ──→ P0/P1
      ├── マルウェア感染 ──→ P0/P1
      ├── 不正アクセス成功 ──→ P1/P2
      └── 不正アクセス試行のみ ──→ P2/P3
      │
      ↓
[インシデント宣言・対応開始]
```

### 4.3 Phase 3: Containment（封じ込め）

#### 4.3.1 短期封じ込め（即時対応）

| 状況 | アクション | 担当 | 所要時間目安 |
|------|----------|------|------------|
| 侵害されたシステム | ネットワーク隔離 | Network Engineer | 15分以内 |
| 悪意あるIP | ファイアウォールでブロック | Network Engineer | 10分以内 |
| 侵害されたアカウント | アカウント無効化 | CSIRT Engineer | 5分以内 |
| 進行中の攻撃 | 該当サービス停止 | CSIRT Engineer | 即時 |

#### 4.3.2 長期封じ込め（安定化）

```
□ システムイメージの取得（フォレンジック用）
□ 証拠保全（Chain of Custody確保）
□ クリーンなバックアップからの仮復旧環境構築
□ 監視強化（侵害インジケータの追加監視）
```

#### 4.3.3 封じ込め判断基準

| 判断項目 | 基準 |
|----------|------|
| ネットワーク遮断 | 横展開の兆候、C2通信確認時 |
| サービス停止 | データ改ざん進行中、大量漏洩進行中 |
| アカウント無効化 | 認証情報漏洩確認時 |
| DNS シンクホール | マルウェアC2ドメイン確認時 |

### 4.4 Phase 4: Eradication（根絶）

#### 4.4.1 根絶作業チェックリスト

```
□ マルウェア・バックドアの完全除去
  - 全システムスキャン実施
  - 不審なファイル・プロセスの削除
  - レジストリ・スタートアップの確認

□ 脆弱性の修正
  - 悪用された脆弱性のパッチ適用
  - 設定不備の修正
  - 不要なサービス・ポートの無効化

□ 認証情報のリセット
  - 侵害されたアカウントのパスワード変更
  - APIキー・トークンの再発行
  - 証明書の失効と再発行

□ 横展開確認
  - 他システムへの影響確認
  - 同一攻撃手法の痕跡調査
```

#### 4.4.2 クリーンアップ手順

1. **感染システムの特定**
   - IOC（Indicators of Compromise）による全システムスキャン
   - ネットワークトラフィック分析

2. **マルウェア除去**
   - 隔離環境での詳細解析
   - 駆除または再構築の判断

3. **システム強化**
   - パッチ適用
   - 設定hardening
   - アクセス制御見直し

### 4.5 Phase 5: Recovery（復旧）

#### 4.5.1 復旧フェーズ

| フェーズ | 内容 | 判断者 |
|----------|------|--------|
| 復旧準備 | バックアップ検証、復旧手順確認 | CSIRT Engineer |
| 段階的復旧 | 重要システムから順次復旧 | CSIRT Team Leader |
| 監視強化期間 | 再発監視、追加IOC監視 | SOC Analyst |
| 通常運用移行 | 監視レベル通常化 | CISO承認 |

#### 4.5.2 復旧優先順位

```
優先度1（即時）: 認証基盤、データベース
優先度2（4時間）: APIサーバー、バックエンド
優先度3（8時間）: フロントエンド、管理画面
優先度4（24時間）: 分析基盤、バッチ処理
```

#### 4.5.3 復旧検証チェックリスト

```
□ システム機能テスト
  - 認証・認可の動作確認
  - 主要機能のE2Eテスト
  - データ整合性確認

□ セキュリティ検証
  - 脆弱性スキャン実施
  - ペネトレーションテスト（簡易）
  - ログ出力確認

□ 監視確認
  - アラート発報テスト
  - ログ収集確認
  - ダッシュボード確認
```

### 4.6 Phase 6: Lessons Learned（教訓）

#### 4.6.1 事後レビュー実施

**開催時期**: インシデントクローズ後1週間以内

**参加者**:
- CSIRT Team Leader（進行）
- 対応に関わった全メンバー
- CISO（P0/P1の場合）

#### 4.6.2 レビュー項目

```
1. タイムライン確認
   - 発生から検知までの時間
   - 検知から封じ込めまでの時間
   - 全体対応時間

2. 対応評価
   - うまくいった点
   - 改善が必要な点
   - 予期せぬ課題

3. 根本原因分析
   - 技術的原因
   - プロセス的原因
   - 人的要因

4. 改善アクション
   - 短期改善（1週間）
   - 中期改善（1ヶ月）
   - 長期改善（四半期）
```

#### 4.6.3 成果物

- インシデントレポート（最終版）
- 改善アクションリスト
- 手順書更新内容
- 訓練計画への反映事項
