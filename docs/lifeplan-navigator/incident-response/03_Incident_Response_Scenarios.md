# LifePlan Navigator インシデント対応シナリオ集

**文書番号**: LPN-IRP-003
**バージョン**: 1.0
**作成日**: 2025-12-11
**承認者**: CSIRT Team Leader
**分類**: 機密

---

## 1. 概要

本文書は、LifePlan Navigatorで発生しうる主要なセキュリティインシデントに対する具体的な対応シナリオを定義する。

### 対象シナリオ
1. データ漏洩インシデント
2. DDoS攻撃
3. ランサムウェア感染
4. 内部不正

## シナリオ1: データ漏洩インシデント

### 1.1 シナリオ概要

**想定状況**: 攻撃者がSQLインジェクション脆弱性を悪用し、ユーザーの個人情報（氏名、メールアドレス、金融情報）を窃取

**重大度**: P0（Critical）

**影響**:
- 個人情報保護法違反リスク
- 顧客信頼の喪失
- 法的責任・損害賠償
- レピュテーション損害

### 1.2 検知ポイント

| 検知源 | 検知内容 | 担当 |
|--------|----------|------|
| WAF | 異常なSQLクエリパターン | SOC Analyst |
| SIEM | 大量のDB読み取りログ | SOC Analyst |
| DBアクティビティログ | 通常外の大量SELECT | CSIRT Engineer |
| 外部通報 | 漏洩データの流通報告 | CSIRT Team Leader |

### 1.3 対応タイムライン

```
[T+0] 検知・アラート発報
      │
      ↓ (15分以内)
[T+15min] 初動対応開始
      - SOC Analyst: アラート確認、影響範囲初期評価
      - エスカレーション: CSIRT Leader, CISO
      │
      ↓ (30分以内)
[T+30min] 封じ込め開始
      - Network Engineer: 攻撃元IPブロック
      - CSIRT Engineer: 脆弱なエンドポイント無効化
      │
      ↓ (1時間以内)
[T+1h] 経営報告・法務連携
      - CISO → CEO報告
      - CLO: 法的義務確認
      │
      ↓ (4時間以内)
[T+4h] 影響範囲確定
      - CSIRT Engineer: フォレンジック分析
      - 漏洩データ・件数特定
      │
      ↓ (24時間以内)
[T+24h] 初期対応完了
      - 脆弱性修正
      - 影響者リスト確定
      │
      ↓ (72時間以内)
[T+72h] 報告・通知
      - 個人情報保護委員会への速報
      - 影響者への通知開始
```

### 1.4 対応手順詳細

#### Phase 1: Identification（識別）

```
□ SOC Analyst
  1. アラート内容確認
     - WAFログで攻撃パターン確認
     - 攻撃元IP、攻撃時刻、対象エンドポイント特定

  2. 影響範囲初期評価
     - アクセスされたテーブル/データ特定
     - 影響ユーザー数の概算

  3. エスカレーション実施
     - P0判定の根拠を添えてCSIRT Leaderに報告
     - Slack #incident-response でインシデント宣言

□ CSIRT Team Leader
  1. インシデント対応本部設置
  2. 役割分担・担当者アサイン
  3. CISO/CEO/CLOへエスカレーション
```

#### Phase 2: Containment（封じ込め）

```
□ Network Engineer
  1. 攻撃元IPのファイアウォールブロック
     $ sudo iptables -A INPUT -s <攻撃IP> -j DROP
     またはWAFルール追加

  2. 該当エンドポイントの一時停止検討
  3. 追加の不審通信監視強化

□ CSIRT Engineer
  1. 脆弱なAPIエンドポイントの緊急停止
  2. 侵害されたアカウントのセッション無効化
  3. DB接続ログの保全
     - 接続元IP、実行クエリ、アクセス時刻

□ App Engineer（支援）
  1. 緊急パッチの準備
  2. 入力値検証の強化案作成
```

#### Phase 3: Eradication（根絶）

```
□ App Engineer
  1. SQLインジェクション脆弱性の修正
     - プリペアドステートメントへの置換
     - 入力値サニタイズ強化
     - WAFルール追加

  2. 同種の脆弱性がないか全エンドポイント確認

□ CSIRT Engineer
  1. 攻撃者が残したバックドア確認
  2. 不審なアカウント・権限の確認
  3. 横展開の痕跡確認
```

#### Phase 4: Recovery（復旧）

```
□ CSIRT Engineer / App Engineer
  1. パッチ適用後のセキュリティテスト
  2. 段階的なサービス再開
  3. 監視強化期間の設定（2週間）

□ SOC Analyst
  1. 攻撃IOCの監視ルール追加
  2. 同一攻撃者の再アクセス監視
```

#### Phase 5: Lessons Learned（教訓）

```
□ CSIRT Team Leader
  1. 事後レビュー会議開催
  2. タイムライン・対応評価
  3. 改善アクション策定

□ 改善項目例
  - セキュアコーディング研修実施
  - SAST/DASTの導入強化
  - DB監視の閾値見直し
```

### 1.5 法的対応（データ漏洩固有）

| アクション | 期限 | 担当 | 報告先 |
|----------|------|------|--------|
| 速報提出 | 発覚から3-5日 | CLO | 個人情報保護委員会 |
| 本人通知 | 速やかに | CMO/CLO | 影響を受けた本人 |
| 確報提出 | 発覚から30日 | CLO | 個人情報保護委員会 |
| 再発防止策報告 | 60日以内 | CISO | 個人情報保護委員会 |

## シナリオ2: DDoS攻撃

### 2.1 シナリオ概要

**想定状況**: 大規模なボットネットによるHTTP Flood攻撃でサービスが応答不能に

**重大度**: P1（High）→ 長時間継続でP0

**影響**:
- サービス停止による機会損失
- 顧客体験の低下
- SLA違反リスク

### 2.2 検知ポイント

| 検知源 | 検知内容 | 担当 |
|--------|----------|------|
| CDN/WAF | リクエスト急増アラート | SOC Analyst |
| APM | レスポンスタイム悪化 | SOC Analyst |
| 監視システム | サービス死活異常 | SOC Analyst |
| 顧客報告 | サービス利用不可 | サポートデスク |

### 2.3 対応タイムライン

```
[T+0] 検知・アラート発報
      │
      ↓ (5分以内)
[T+5min] 攻撃確認・初動
      - SOC: トラフィック分析
      - Network Engineer: 緊急呼び出し
      │
      ↓ (15分以内)
[T+15min] 緩和措置開始
      - CDN/WAFのレートリミット強化
      - 攻撃パターンのブロックルール追加
      │
      ↓ (30分以内)
[T+30min] 追加対策
      - 上流ISP/CDNへの連絡
      - ジオブロッキング検討
      │
      ↓ (1時間)
[T+1h] 状況評価
      - 緩和効果確認
      - エスカレーション判断
      │
      ↓ (攻撃終息後)
[T+end] 事後対応
      - ログ分析
      - 防御策強化
```

### 2.4 対応手順詳細

#### Phase 1: Identification（識別）

```
□ SOC Analyst
  1. 攻撃タイプの特定
     - Volumetric（帯域消費型）
     - Protocol（プロトコル攻撃）
     - Application Layer（L7攻撃）

  2. 攻撃規模の把握
     - トラフィック量（Gbps/Mpps）
     - リクエスト数（req/sec）
     - 攻撃元IP数

  3. 攻撃元の分析
     - 地理的分布
     - ASN分布
     - ボットネット特性

□ Network Engineer
  1. インフラ状況確認
     - サーバー負荷
     - ネットワーク帯域使用率
     - CDN/WAFステータス
```

#### Phase 2: Containment（封じ込め）

```
□ Network Engineer
  1. CDN/WAF対策
     - レートリミット強化
     - チャレンジ（CAPTCHA）有効化
     - 攻撃シグネチャのブロックルール追加

  2. ネットワーク層対策
     - ACLによる攻撃トラフィックフィルタ
     - BGP Blackholing（最終手段）

  3. 外部連携
     - CDNプロバイダーへの緊急連絡
     - ISPへのアップストリーム対策依頼

□ CSIRT Team Leader
  1. 状況報告（30分毎）
  2. サービス縮退運転の判断
  3. 外部DDoS防御サービス発動判断
```

#### Phase 3: 攻撃中の監視・調整

```
□ SOC Analyst
  1. リアルタイム監視
     - 攻撃パターンの変化検知
     - 緩和策の効果測定

  2. ログ収集
     - 全トラフィックログ保全
     - 攻撃元IP・パターン記録

□ Network Engineer
  1. 緩和策の微調整
     - 誤検知による正規ユーザーブロックの解消
     - 新しい攻撃パターンへの対応
```

#### Phase 4: Recovery（復旧）

```
□ Network Engineer
  1. 攻撃終息の確認
     - トラフィック正常化確認
     - サービス応答確認

  2. 緊急対策の段階的解除
     - レートリミットの緩和
     - ブロックルールの精査

  3. サービス正常化確認
     - 全機能の動作確認
     - パフォーマンス確認

□ SOC Analyst
  1. 監視継続（攻撃再発に備える）
  2. 24時間の強化監視期間設定
```

#### Phase 5: Lessons Learned（教訓）

```
□ 改善項目例
  - DDoS防御サービスの導入/強化
  - オートスケーリング設定の見直し
  - レートリミットの基準値調整
  - Runbook（対応手順書）の更新
```

### 2.5 DDoS固有の判断基準

| 状況 | 判断 | 承認者 |
|------|------|--------|
| 攻撃30分継続、サービス影響継続 | 外部DDoS防御サービス発動 | CSIRT Leader |
| 攻撃1時間継続、緩和効果なし | CDN緊急増強/ISP連携 | CISO |
| 攻撃2時間継続 | P0昇格、経営報告 | CISO |
| 正規ユーザーブロック発生 | ルール即時緩和 | CSIRT Leader |

## シナリオ3: ランサムウェア感染

### 3.1 シナリオ概要

**想定状況**: フィッシングメール経由でランサムウェアが感染、重要ファイルが暗号化され身代金要求

**重大度**: P0（Critical）

**影響**:
- 業務停止
- データ喪失リスク
- 身代金要求への対応判断
- 法的・コンプライアンス問題

### 3.2 検知ポイント

| 検知源 | 検知内容 | 担当 |
|--------|----------|------|
| EDR | 不審なファイル暗号化挙動 | SOC Analyst |
| SIEM | 大量のファイル変更イベント | SOC Analyst |
| ユーザー報告 | ファイルが開けない、脅迫画面 | サポートデスク |
| バックアップシステム | 急激なデータ変更 | CSIRT Engineer |

### 3.3 対応タイムライン

```
[T+0] 検知・アラート発報
      │
      ↓ (即時)
[T+0-5min] 緊急隔離
      - 感染端末のネットワーク切断
      - 横展開防止
      │
      ↓ (15分以内)
[T+15min] 影響範囲特定
      - 感染端末の特定
      - 暗号化されたデータの確認
      │
      ↓ (30分以内)
[T+30min] 経営報告
      - CEO/CISO/CLO/CFO報告
      - 対応方針協議
      │
      ↓ (1時間以内)
[T+1h] 対応方針決定
      - 身代金対応判断
      - バックアップからの復旧判断
      │
      ↓ (4-24時間)
[T+4-24h] 復旧作業
      - クリーンな環境構築
      - バックアップからのリストア
      │
      ↓ (1週間)
[T+1w] 完全復旧・強化
      - 全システム復旧
      - 再発防止策実施
```

### 3.4 対応手順詳細

#### Phase 1: Identification（識別）

```
□ SOC Analyst
  1. アラート確認
     - EDRアラートの詳細確認
     - マルウェアファミリーの特定

  2. 感染経路の初期調査
     - 不審なメール/添付ファイル
     - 不正なダウンロード
     - 外部デバイス

□ CSIRT Engineer
  1. マルウェア検体の取得
  2. 既知の復号化ツール有無確認
  3. 脅威インテリジェンス照合
```

#### Phase 2: Containment（封じ込め）- **最優先**

```
□ Network Engineer - 即時実施
  1. 感染端末のネットワーク隔離
     - ネットワークケーブル抜線
     - WiFi無効化
     - VLANによる隔離

  2. 横展開防止
     - セグメント間通信制限
     - SMB/RDP等の一時的ブロック
     - 管理者アカウントのパスワードリセット

  3. C2通信ブロック
     - ファイアウォールでC2ドメイン/IPブロック
     - DNSシンクホール設定

□ CSIRT Engineer
  1. 全端末のEDRスキャン実行
  2. 感染端末の特定・リスト化
  3. 重要サーバーの保護強化
```

#### Phase 3: Eradication（根絶）

```
□ CSIRT Engineer
  1. 感染端末の調査
     - フォレンジックイメージ取得
     - マルウェア挙動分析
     - 永続化メカニズム特定

  2. 感染経路の特定
     - Initial Access（初期侵入）の特定
     - Execution（実行）方法の特定

  3. 残存脅威の排除
     - 他のバックドア確認
     - 侵害されたアカウントの特定

□ App Engineer（支援）
  1. 悪用された脆弱性の修正
  2. メール/Web経由の感染防止強化
```

#### Phase 4: Recovery（復旧）

```
□ CSIRT Team Leader
  1. 復旧方針決定
     - バックアップからの復旧
     - システム再構築
     - 身代金支払いは原則非推奨

  2. 復旧優先順位決定
     - 基幹業務システム優先
     - 段階的復旧計画

□ CSIRT Engineer / Network Engineer
  1. クリーン環境での復旧
     - 新規構築またはクリーンイメージからの復元
     - バックアップデータの検証（感染していないことを確認）

  2. セキュリティ強化の上で復旧
     - パッチ適用
     - EDR設定強化
     - ネットワークセグメンテーション強化

□ SOC Analyst
  1. 復旧後の監視強化
  2. 再感染の兆候監視
```

### 3.5 身代金対応判断

| 判断要素 | 確認事項 | 担当 |
|----------|----------|------|
| バックアップ | 有効なバックアップの存在確認 | CSIRT Eng |
| 復号化ツール | 公開されている復号化ツールの有無 | CSIRT Eng |
| 攻撃者 | 攻撃者グループの信頼性（支払い後復号する傾向か） | CTI Analyst |
| 法的リスク | 制裁対象への支払いとならないか | CLO |
| 財務影響 | 身代金額 vs 復旧コスト vs 事業損失 | CFO |

**原則**: 身代金支払いは非推奨（支払いが必ずしも復旧を保証しない、犯罪組織への資金提供）

### 3.6 法執行機関との連携

```
□ CLO担当
  1. 警察（サイバー犯罪対策課）への届出
  2. JPCERT/CCへの報告
  3. 業界ISACへの情報共有（任意）
```

## シナリオ4: 内部不正

### 4.1 シナリオ概要

**想定状況**: 退職予定の従業員が顧客データを大量にダウンロードし、競合他社に持ち出そうとしている

**重大度**: P1（High）→ 持ち出し確認でP0

**影響**:
- 顧客データ漏洩
- 競争上の損害
- 法的責任
- 内部統制の信頼性

### 4.2 検知ポイント

| 検知源 | 検知内容 | 担当 |
|--------|----------|------|
| DLP | 大量データダウンロード | SOC Analyst |
| UEBA | 異常なユーザー行動 | SOC Analyst |
| アクセスログ | 業務外データアクセス | CSIRT Engineer |
| 内部通報 | 同僚からの報告 | HR/CISO |

### 4.3 対応タイムライン

```
[T+0] 検知・内部通報
      │
      ↓ (即時)
[T+0-30min] 秘密裏に調査開始
      - 対象者に気づかれないよう調査
      - 監視強化（合法的範囲で）
      │
      ↓ (1-4時間)
[T+1-4h] 証拠収集
      - アクセスログ保全
      - ダウンロードデータ特定
      - メール/チャット調査（法務確認済み）
      │
      ↓ (証拠確保後)
[T+x] アクセス停止・聴取
      - 対象者のアクセス権限停止
      - HR/法務同席での聴取
      │
      ↓ (事実確定後)
[T+y] 処分・再発防止
      - 懲戒処分
      - 法的措置検討
      - 統制強化
```

### 4.4 対応手順詳細

#### Phase 1: Identification（識別）- **秘密厳守**

```
□ SOC Analyst
  1. アラート検証
     - 誤検知の可能性排除
     - 行動パターンの分析

  2. 初期情報収集
     - 対象者の役職・権限
     - 退職予定等の人事情報（HR経由）

□ CSIRT Team Leader
  1. 調査チーム編成（最小限の人数）
  2. 情報漏洩防止（調査自体の秘密保持）
  3. CLO/HR責任者のみに報告
```

#### Phase 2: Investigation（調査）

```
□ CSIRT Engineer
  1. アクセスログ詳細分析
     - アクセスしたデータの特定
     - ダウンロードしたファイルの特定
     - アクセス時刻・頻度

  2. データ持ち出し経路の調査
     - USB等外部デバイス使用
     - クラウドストレージへのアップロード
     - メール添付での送信
     - 印刷

  3. フォレンジック証拠保全
     - 対象者PCのイメージ取得（可能であれば）
     - ログの完全性確保

□ CLO
  1. 調査の法的妥当性確認
     - プライバシーへの配慮
     - 監視の適法性
  2. 証拠の法的有効性確保
```

#### Phase 3: Containment（封じ込め）

```
□ Network Engineer / CSIRT Engineer
  1. 段階的なアクセス制限
     - 機密データへのアクセス制限
     - ネットワーク監視強化

  2. データ持ち出し防止
     - DLPルール強化
     - USBポート無効化
     - クラウドアップロード制限

※ 対象者に気づかれないよう慎重に実施
```

#### Phase 4: Action（対処）

```
□ HR / CLO / CISO
  1. 対象者への対応
     - アクセス権限即時停止
     - 社内聴取（HR/法務同席）
     - 退職済みの場合は法的措置検討

  2. 処分決定
     - 懲戒処分（就業規則に基づく）
     - 損害賠償請求の検討
     - 刑事告訴の検討

□ CSIRT Team Leader
  1. 漏洩データの範囲確定
  2. 影響を受けた顧客への通知判断
  3. 関係者への報告（最小限）
```

#### Phase 5: Lessons Learned（教訓）

```
□ 改善項目例
  - Need to Know原則の徹底
  - DLP/UEBA導入・強化
  - 退職者のオフボーディングプロセス強化
  - 定期的な権限棚卸
  - セキュリティ意識向上教育
```

### 4.5 内部不正固有の注意点

| 項目 | 注意点 |
|------|--------|
| **秘密保持** | 調査中は対象者・周囲に気づかれないよう徹底 |
| **法的配慮** | 監視・調査は法令・就業規則の範囲内で実施 |
| **証拠保全** | 後の法的手続きに備え、証拠の完全性を確保 |
| **推定無罪** | 調査確定まで処分を決定しない |
| **報告範囲** | 知る必要がある人のみに限定 |

### 4.6 役員・経営層関与の場合

```
通常のエスカレーションラインを使用せず:

[検知] → [外部顧問弁護士] → [監査役/取締役会]

※ 当該役員の上司・同僚を経由しない
```
